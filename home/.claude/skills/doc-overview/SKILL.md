# プロジェクト概要・用語集 作成・更新

現在の作業コンテキストに基づき、プロジェクト概要または用語集を作成・更新する。
`/doc` から委譲された場合も、直接呼び出された場合もこのファイルの手順に従う。

## ステップ1: ディレクトリ準備と既存ファイルの確認

```bash
# 配置先ディレクトリの存在確認。なければ作成する
ls docs/00_overview/ 2>/dev/null || mkdir -p docs/00_overview/

# 既存ファイルを確認（更新対象か新規作成かを判断）
ls docs/00_overview/
```

既存ファイルが対象内容をカバーしている場合は**新規作成せず更新**する。

## ステップ2: ドキュメントの作成・更新

以下の品質原則を制約として守りながら作成する。

| 原則 | 内容 |
|------|------|
| Single Purpose | プロジェクト概要・目的・用語集のみを記述する。要件詳細や実装詳細は含まない |
| ADR-Based Decisions | 大きな方針変更の背景は ADR に記録済みか確認する |
| Single Source of Truth | 要件・仕様の詳細は各専門ドキュメントに委ねる。概要は概要にとどめる |
| Template-Driven Creation | 以下のテンプレートに沿って作成し、実際の内容で埋める |
| シークレット不在 | 環境変数・パスワード・トークン等をリポジトリに記載しない |

**ファイル名（概要）**: `docs/00_overview/overview.md`
**ファイル名（用語集）**: `docs/00_overview/glossary.md`

**概要テンプレート**:
```markdown
# <プロジェクト名> 概要

## 目的

（このプロジェクトが何を解決するか。1〜3行で記述）

## スコープ

（対象範囲と対象外範囲を明記）

## 主要コンポーネント

- **<コンポーネント名>**: （役割の一行説明）

## 関連ドキュメント

- [要件定義](../10_requirements/)
- [機能仕様](../20_specs/)
- [設計](../30_design/)
```

**用語集テンプレート**:
```markdown
# 用語集

| 用語 | 定義 |
|------|------|
| <用語> | <定義> |
```

## ステップ3: 品質自己チェック

| チェック項目 | 確認内容 |
|-------------|---------|
| Single Purpose | プロジェクト概要・用語のみで、要件や実装詳細が混在していないか |
| ADR-Based Decisions | 方針変更があれば対応する ADR が存在するか |
| Single Source of Truth | 詳細情報は各専門ドキュメントにリンクし、重複記述していないか |
| Template-Driven | テンプレート構成に沿い、プレースホルダーが残っていないか |
| 配置先ディレクトリ | `docs/00_overview/` に配置されているか |
| 命名規則 | kebab-case か |
| シークレット不在 | 機密情報が含まれていないか |
| Mermaid 形式 | 図がある場合、画像ファイルではなく Mermaid で記述されているか |

チェック結果は「✅ 全項目パス」または「⚠️ 修正済み（修正内容）」と明記する。

## ステップ4: 報告

- 作成・更新したファイルのパスをユーザーに報告する
- ステップ3のチェック結果を報告する

## 注意事項

- 詳細な要件・仕様・設計は概要ドキュメントに記述せず、それぞれの専門ドキュメントにリンクする
- 作成するドキュメントは現在の作業コンテキストに即した具体的な内容にする（テンプレートをそのまま出力しない）
